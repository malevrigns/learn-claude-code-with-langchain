# learn-claude-code-with-langchain

## 一份面向未来的任务总述

这个仓库不是为了记录我现在手上的一个 MVP 做了什么，而是为了明确我接下来要做什么。

它的直接思想来源是 [shareAI-lab/learn-claude-code](https://github.com/shareAI-lab/learn-claude-code)。真正打动我的，不是某几个具体脚本，也不是某个表层交互，而是它背后的判断：Agent 的关键并不只是模型本身，更在于我们如何为模型构造一个可工作的环境。这个环境不是装饰品，不是把几个工具硬接到模型后面就算完成，它本身就是 Agent 工程的核心。

因此，这个仓库会以 LangChain 为主要技术路径，逐步重构一套我理解中的 Agent Harness。当前代码只是起点，这份 README 才是后续工作的方向说明。

## 我对 Agent 的理解

在我看来，Agent 不是一个“会聊天的模型”，也不是一个“能调用几个工具的脚本”。如果一个系统只能回答问题，却无法持续观察环境、理解目标、采取行动、接受反馈并调整后续行为，那么它更像一个带接口的问答器，而不是一个真正意义上的 Agent。

我理解的 Agent，至少要同时具备四个层面：

1. `目标`
   Agent 不是被动响应，它需要围绕任务目标持续组织自己的行为。
2. `感知`
   它必须能够看到环境状态，读取外部信息，理解当前世界发生了什么。
3. `行动`
   它需要通过工具、接口或执行环境去改变外部世界，而不仅仅是生成文字。
4. `反馈闭环`
   它的行动会产生结果，这些结果要重新进入上下文，成为下一步决策的依据。

如果把这个过程进一步压缩，我对 Agent 的理解可以写成这样：

```text
Agent = 具备目标导向能力的模型 + 可感知、可行动、可反馈的环境
```

这也是为什么我非常认同上游仓库所强调的 `Model + Harness` 视角。模型当然重要，因为真正的泛化能力、抽象能力、规划能力、语言能力，主要来自模型本身的预训练与后训练；但如果没有一个设计良好的 Harness，模型的这些能力就无法被稳定释放出来。

也就是说，Agent 的“智能”并不是工程师通过无数 if-else 写出来的。工程师真正要做的，是构造一个世界，让模型在这个世界里工作。这个世界要告诉模型：

- 它现在在哪
- 它可以看到什么
- 它可以调用什么能力
- 哪些动作是安全的，哪些动作需要限制
- 执行结果会以什么形式返回
- 长任务如何被拆解、恢复和持续推进

所以，Agent 工程在很大程度上不是“替模型思考”，而是“为模型组织现实”。

## Agent 的发展

如果回看最近几年的演进，我认为 Agent 的发展并不是一条简单的“模型越来越强”的直线，而是一个从“语言接口”逐渐走向“环境接口”的过程。

### 第一阶段：聊天模型阶段

最早的大模型产品，本质上是语言交互系统。它们最擅长的是理解问题、组织回答、生成内容。在这个阶段，人们讨论的是 Prompt Engineering，是如何提问、如何约束输出、如何设计角色。

这一阶段的价值很大，因为它让人们第一次直观感受到模型具备某种通用推理与表达能力。但它的边界也很明显：模型只是在“说”，并没有真正“做”。

### 第二阶段：工具增强阶段

当模型开始接入搜索、计算器、数据库、代码执行器、浏览器之后，情况开始变化。模型不再只是根据参数记忆回答问题，而是可以调用外部系统，把世界中的真实信息和真实动作纳入自己的工作流程。

这一步很关键，因为它让 Agent 从纯语言系统变成了“语言 + 行动”的系统。模型第一次拥有了手脚。

但在这个阶段，很多实现仍然比较浅。它们虽然能调用工具，却往往只是把工具调用当成一次性的能力外挂，缺少持续任务管理、上下文治理、权限边界和复杂环境中的执行组织。

### 第三阶段：工作流伪装成 Agent 的阶段

随着 Agent 概念流行，大量系统开始出现。很多产品看起来像 Agent，实际上只是预先写好的工作流：节点 A 之后一定去节点 B，工具 X 调完一定接工具 Y，错误出现就走固定分支。

这种系统在一些明确任务上当然有价值，但它的本质更接近自动化流程，而不是具备开放环境适应能力的 Agent。它们往往依赖工程师提前枚举路径，一旦问题超出设计范围，系统就会迅速脆弱。

这一阶段让我越来越确信一个判断：真正的 Agent 不能主要依赖工程师事先把“思维过程”全部写死。工程师可以给它边界、工具和结构，但不应该用流程图代替模型本身的判断。

### 第四阶段：Harness Engineering 阶段

我认为，这才是 Agent 真正开始走向工程化的阶段。

在这个阶段，问题不再是“如何让模型多说一步”，而是：

- 如何让模型在长任务里保持连续性
- 如何让模型按需获取知识，而不是被上下文污染
- 如何让模型安全地操作本地环境和外部系统
- 如何把复杂任务拆分给不同执行单元
- 如何在并行任务中保持隔离
- 如何记录轨迹、回放过程、做评估和持续改进

换句话说，工程重心从 Prompt Engineering 转向了 Harness Engineering。Agent 的表现，不再只是由提示词决定，而是越来越取决于工具系统、上下文系统、权限系统、任务系统和执行环境。

### 第五阶段：从单体 Agent 走向系统级 Agent

再往前走，Agent 很可能不再只是“一个模型 + 一组工具”，而会变成一个真正的软件系统：

- 有主 Agent，也有子 Agent
- 有任务拆分，也有依赖编排
- 有同步调用，也有异步消息
- 有共享目标，也有上下文隔离
- 有工作区，也有权限边界
- 有执行日志，也有评估闭环

到了这个阶段，Agent 已经不只是“一个功能”，而更像一种新的软件形态。它不是简单替代传统程序，而是在很多开放式、高不确定性的任务里，成为连接模型能力与现实世界的执行层。

## 为什么 Coding Agent 是一个理想切入口

如果要真正理解 Agent，Coding Agent 是非常好的训练场。

原因不是“写代码这件事更酷”，而是因为编程任务天然具备非常清晰的反馈结构：

- 可以读文件
- 可以改文件
- 可以运行命令
- 可以看到报错
- 可以查看 diff
- 可以运行测试
- 可以不断验证修改是否有效

这意味着，Coding Agent 天然适合构建感知、行动、反馈的闭环。它不像一些抽象任务那样难以验证，也不像纯对话那样缺少真实环境约束。对于我来说，先从 Coding Agent 入手，不是为了把问题做窄，而是为了在一个反馈足够强的场景里把 Harness 打磨清楚。

## 这也是我接下来要做的工作

基于上面的理解，我接下来的任务不是简单地“用 LangChain 重写一个脚本”，而是要用 LangChain 去重构一套更完整的 Agent Harness。

我真正要做的事情，大致有三层。

### 第一层：重建最基本的 Agent 闭环

这一层的目标，是让系统先具备最小但真实的 Agent 结构：

- 模型能够接收目标
- 模型能够观察本地环境
- 模型能够调用工具采取行动
- 工具结果能够稳定回流到上下文
- 系统能够在多轮执行中持续推进任务

这一步不是终点，但它是后面所有能力的基础。

### 第二层：把 Harness 做厚

如果只有一个模型和一个 shell 工具，这仍然只是演示。真正的重构会继续向下铺设基础设施：

- 文件读取、写入、编辑、搜索
- Git 状态读取、差异查看、提交辅助
- 命令执行的权限约束
- 文档、规范、代码知识的按需引入
- 上下文压缩、摘要与长期状态保存
- 更好的错误处理、日志与可观测性

这一层的目标，是让 Agent 从“能动一下”变成“能够持续工作”。

### 第三层：让系统具备长期演进能力

真正有价值的 Agent 工程，不会停在单轮工具调用。后续我会继续把它推向更复杂的形态：

- 任务拆分
- 子 Agent 委派
- 异步消息或邮箱机制
- 隔离工作区中的并行执行
- 审批流与权限治理
- tracing、回放、评估与优化

这一步的意义在于，把 Agent 从一个 demo 提升成一个能够扩展、能够维护、能够持续优化的系统。

## 为什么我选择 LangChain

我选择 LangChain，不是因为它能自动解决 Agent 的所有问题，而是因为它适合作为重构过程中的中间层与组织层。

它至少提供了几个现实价值：

- 比较统一的模型接口
- 更规范的消息对象
- 更自然的工具定义方式
- 更方便扩展到复杂状态机或 LangGraph 工作流

对我来说，LangChain 的意义不是“帮我偷懒写 Agent”，而是帮助我把这套 Harness 拆得更清楚、接得更稳定、演进得更自然。

## 我的工作方法

这份仓库后续的推进，会尽量遵守下面这些原则：

- 不用流程图去冒充智能
- 不用硬编码去替代模型判断
- 把工程重点放在工具、上下文、权限、状态与执行环境
- 让知识按需进入，而不是一次性堆满上下文
- 让工具原子化、可组合、可审计
- 让系统默认隔离、默认最小权限
- 让重要过程可追踪、可回放、可评估
